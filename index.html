🧠 MVVM (Model-View-ViewModel)

The MVVM (Model-View-ViewModel) design pattern enforces a clear separation of concerns by decoupling the user interface (View) from the underlying business logic and data (Model), with the ViewModel acting as a mediator. This structure enhances code clarity, testability, and long-term maintainability.

+-------------+       +----------------+       +--------+
|   View      | <-->  |  ViewModel     | <-->  | Model  |
+-------------+       +----------------+       +--------+
  • Model: Application logic and data
  • View: User interface, data binding
  • ViewModel: Connection between UI and data

🔁 Key Benefits

  • Two-way data binding
  • Improved maintainability
  • Separation of concerns
  • Facilitated unit testing

🕵️‍♂️ Privacy Tools

If you're concerned about privacy or internet censorship, consider using open-source proxy technologies such as Shadowsocks-libev on the server side and v2rayN as a Windows client.

🔐 Architecture

        [v2rayN - Client]
         +-------------+
         |  (Windows)  |
         | Shadowsocks |
         |  Protocol   |
         +-------------+
                |
(Encrypted traffic over Internet)
                |
         +-------------+
         | (UbuntuVPS) |
         | Shadowsocks |
         |   libev     |
         +-------------+
  • Client: v2rayN uses Shadowsocks protocol to encrypt traffic before sending it.
  • Transport: The encrypted traffic (e.g. AES-256-GCM, xchacha20-ietf-poly1305) passes through the internet.
  • Server: shadowsocks-libev decrypts the traffic and forwards it to the destination (e.g. Google, GitHub).
